# 通用代码生成纪律

适用范围：当前仓库。

## 1. 元原则

- **结论优先，细节后置**：结论和关键信息放最前面。
- **没有模糊空间**："必须"、"禁止"无任何例外。
- **量化标准优先**：规则中的数字（如超过 5 个文件、修复 2 次）是硬停止点，不主观放宽。
- **代码极简优先**：能一行写完不写多行；不新增未显式要求的抽象；不引入可避免的新依赖；减少防御式写法与无意义包装。
- **调研先于方案，方案先于编码**：未读相关 skill、未确认信息充足、未消除隐藏假设，不得进入方案设计；未确认方案不得进入编码。

## 2. 最小必要实现原则

动手前先理解问题，阅读相关代码并追踪真实流程。

按以下阶梯选择实现方式，只有上一级不满足时才进入下一级：

1. 不需要存在 → 不写（YAGNI）。
2. 已有代码可以复用 → 复用。
3. 标准库可以实现 → 用标准库。
4. 原生平台特性可以实现 → 用原生特性。
5. 已安装依赖可以实现 → 用已安装依赖。
6. 以上都不行 → 只写最小必要实现。

安全、验证、错误处理、数据完整性、可访问性不可省略。

## 3. 防御代码边界

- 信任框架与类型系统：框架已校验、前置检查或 schema 已保证的约束，不在业务代码中重复防御。
- 不"写保险"：只在明确需要时做类型/空值/边界判断，不为"万一"堆叠防御。
- 不简单包装：不创建无新增语义的变量别名、函数套壳或同义转发。
- 不提前抽象：简单逻辑、单次使用、调用 ≤ 2 处的片段直接内联；只在真实重复出现且边界稳定时封装。
- 可访问性、安全、验证、错误处理、数据完整性不是防御代码，不可省略。

## 4. 交互纪律

- 字母开头（A/B/C）→ 执行方案；数字开头（1/2/3）→ 执行建议；其余回复视为讨论。
- 需求明确、无歧义、无架构冲突时，输出精确范围清单，等用户确认；未确认或用户提出异议时，重新分析并再次确认。
- 需求模糊、缺参数、有歧义、与现有架构冲突 → 输出澄清问题，等回答。
- 需要方案选择时，必须输出 3 个方案（A / B / C），等字母确认。
- 执行中偏离原定方案 → 停止，输出新方案，等确认。

## 5. 分析纪律

- 分析必须结构化：问题拆解、影响面、3 个方案对比（核心思路、改动范围、优势、风险、工作量、决策属性）、建议。
- 每个方案回答主路径与异常路径；复杂任务额外回答回退策略。

## 6. 执行纪律

- 确认方案后输出精确范围清单（修改哪些、绝不碰哪些）。
- 涉及 ≥3 个文件时声明改动顺序。
- 必须扩展范围时立即停止，重新分析等确认；执行中遇到未预期的新依赖、新文件、新接口，同样必须停止，不得顺手修改。
- 删除文件、改核心配置、改公共接口、批量替换/删除前标注[破坏性操作]。
- 每改完一个文件自检风格、无未使用导入、无调试残留、不破坏现有功能、边界已处理。
- 改完后运行 check / lint / 编译 fallback，输出验证结果。
- 同一个问题修复 2 次仍未通过，立即停止并输出失败终止报告。
- 完成后逆向检查：有无超范围改动、多余依赖/抽象、未清理临时内容。

## 7. Skill 使用规则

- 优先根据 skill 描述自动推断并调用全局 skills（`~/.agents/skills/`）。
- 匹配到 skill 后，先完整阅读并遵循该 skill 规范，再开始设计和实现。
- 代码生成时不重复 skills 已定义的内容，不自行发明与 skill 冲突的命名或约定。

## 8. 通用禁止项

- 禁止在信息不足、需求模糊、方案未确认时直接执行代码修改。
- 禁止隐藏假设或用猜测代替调研。
- 禁止擅自扩展或缩减已确认的方案范围。
- 禁止在配置/脚本中写入绝对路径。
- 禁止用主观判断绕过量化标准。
- 禁止擅自修改项目版本号或执行发布/部署流程，除非用户明确确认。

## 9. 项目规范入口

本项目具体技术栈、模块边界、业务约束见仓库根目录 `AGENTS.md` 及子目录 `AGENTS.md`。

## 10. 交付标准

一次改动视为完成，必须同时满足：

1. 范围清单中的所有项已处理，清单外无任何改动。
2. 所有相关文件无新增错误、无新增警告（允许与本次改动无关的已有警告保持不变）。
3. 若存在类型检查、编译或测试入口，必须执行并通过最小相关验证；无法通过时按"同一个问题修复 2 次仍未通过"处理。
4. 无临时文件、无调试代码、无未使用依赖、无未使用的导入或变量。
