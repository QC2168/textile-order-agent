# 纺织订单协同 Agent Demo 设计

日期：2026-07-11

## 目标

基于当前 `app.py` 做一个 Chainlit 路演 Demo，展示一个纺织订单协同 Agent 如何通过 mock 数据、function call 和提示词完成三类核心能力：

1. 订单全流程追踪
2. 异常提醒
3. 客户沟通自动化

Demo 面向黑客松或路演，不接真实 ERP、MES、企微、短信、邮件或数据库。重点是证明“LLM + 工具调用 + 领域提示词”在纺织订单协同场景中可行。

## 当前项目上下文

当前仓库已有一个 Chainlit 单文件入口 `app.py`：

- 使用 `AsyncOpenAI` 连接 ModelScope 腾讯混元 Hy3。
- 已有基础聊天事件：`on_chat_start`、`on_message`。
- 已预留 OpenAI Function Calling 格式的工具配置说明。
- 当前还没有真实工具调用循环、mock 订单数据或订单领域提示词。

项目中另有 Next.js ColorBridge MVP，但本 Demo 明确不依赖也不修改 `web/` 目录。Chainlit `app.py` 是本次唯一应用入口。

## 范围

### 包含

- 在 `app.py` 中加入纺织订单协同 Agent 的系统提示词。
- 在 `app.py` 中加入 2-3 条 mock 纺织订单数据。
- 定义 OpenAI-compatible function tools。
- 实现本地工具执行函数。
- 实现最小 tool-calling 循环：模型请求工具、后端执行工具、模型基于工具结果生成最终回复。
- 更新 Chainlit 欢迎语，提供可直接复制的演示问题。

### 不包含

- 不做真实 ERP/MES/企微/短信/邮件集成。
- 不做登录、权限、多租户。
- 不做持久化数据库。
- 不修改 `web/`、Prisma、SQLite 或 ColorBridge MVP。
- 不做生产部署。
- 不生成真实客户消息，只生成“建议话术”。

## 核心用户体验

用户进入 Chainlit 后看到欢迎语和示例问题，例如：

- “帮我看一下订单 BS-20260711-001 当前进度。”
- “扫描今天有哪些订单有延期风险。”
- “订单 BS-20260711-002 客户一直没确认色样，帮我生成一段客户沟通话术。”

Agent 回复必须结论优先，优先展示：

- 当前订单状态
- 风险或异常
- 建议动作
- 可直接发送给客户的沟通话术

如果用户询问不存在的订单，Agent 必须说明未找到，不能编造订单状态。

## Mock 数据

内置 3 类订单：

1. 正常订单
   - 已完成客户确认和打样。
   - 正在生产或待质检。
   - 无明显延期风险。

2. 风险订单
   - 客户未确认色样或关键字段缺失。
   - 交期接近，存在延期风险。
   - 需要跟单员催确认。

3. 异常订单
   - 打样不通过或质检色差超阈值。
   - 需要内部复核并通知客户预计影响。
   - 需要生成解释性客户话术。

每个订单至少包含：

- 订单号
- 客户名称
- 面料品类
- 颜色需求
- 数量
- 交期
- 当前阶段
- 负责人
- 时间线事件
- 风险标签
- 下一步建议

## Function Tools

### `get_order_summary(order_id)`

查询单个订单摘要。

返回：

- 客户
- 面料
- 颜色需求
- 数量
- 交期
- 当前阶段
- 当前结论

主路径：订单存在时返回结构化摘要。  
异常路径：订单不存在时返回明确的 not found 结果。

### `get_order_timeline(order_id)`

查询订单全流程时间线。

返回：

- 下单
- 需求确认
- 打样
- 客户确认
- 生产
- 质检
- 出货

主路径：返回已完成、进行中、阻塞或未开始的节点。  
异常路径：订单不存在时返回 not found。

### `detect_order_risks(order_id=None)`

识别单个订单或全部订单风险。

返回：

- 风险等级
- 风险类型
- 触发原因
- 建议动作

主路径：指定订单时只分析该订单；未指定订单时扫描全部 mock 订单。  
异常路径：订单不存在时返回 not found。

### `draft_customer_message(order_id, scenario)`

生成客户沟通话术所需的上下文数据。

返回：

- 客户名称
- 订单背景
- 当前问题
- 建议沟通目标
- 注意事项

LLM 基于工具返回内容生成最终中文话术。话术必须克制、专业、可直接发送。

### `advance_order_stage(order_id, stage)`

可选工具，用于路演时模拟推进订单阶段。

主路径：允许从 mock 状态中更新当前阶段并追加时间线事件。  
异常路径：订单不存在或阶段非法时返回错误说明。

如果实现时间紧张，该工具可以不做，不影响核心演示。

## 提示词策略

系统提示词定义 Agent 为“纺织订单协同助理”，职责是帮助跟单员追踪订单、发现风险、准备客户沟通内容。

必须约束：

- 查询订单状态前必须调用工具。
- 不得凭空编造订单、客户、进度或异常。
- 回复先给结论，再给依据。
- 风险提示必须给出触发原因和建议动作。
- 客户话术必须像真实跟单员发送给客户的内容，避免夸大承诺。
- 对 mock 数据之外的问题，可以解释当前 Demo 数据范围。

## Tool Calling 流程

1. Chainlit 收到用户消息。
2. 后端将系统提示词、历史消息和 tools 传给模型。
3. 如果模型返回 `tool_calls`，后端执行对应本地函数。
4. 后端把工具结果作为 `role=tool` 消息追加回上下文。
5. 再次调用模型生成面向用户的最终回复。
6. Chainlit 显示工具调用步骤和最终回答。

主路径使用 OpenAI-compatible tool calling。

兼容性回退：

- 如果 ModelScope Hy3 的 tool calling 响应不稳定，保留一个规则路由 fallback。
- 规则路由根据关键词和订单号直接调用本地工具，再让模型只负责组织最终回复。
- 这样即使模型工具调用不可用，路演仍能完成订单追踪、异常提醒和客户话术生成。

## 错误处理

- 订单不存在：明确提示“当前 mock 数据未找到该订单”。
- 工具参数缺失：要求用户补充订单号或场景。
- 模型调用失败：返回可读错误，并提示可以重试。
- 工具调用失败：展示失败原因，不吞错。
- tool calling 不兼容：切换到规则路由 fallback。

## 可行性

可行性高。

原因：

- 当前 `app.py` 已经具备 Chainlit 消息处理和 OpenAI-compatible client。
- Chainlit 支持会话状态、消息、Action 和 Step，足够支撑路演 Demo。
- OpenAI Python SDK 支持 `tools`、`tool_choice`、`tool_calls` 和 `role=tool` 消息循环。
- mock 数据不需要外部系统依赖，演示稳定性可控。

主要不确定性是 ModelScope Hy3 对 OpenAI tool calling 的兼容程度。该风险可以通过规则路由 fallback 控制。

## 开发成本

### 路演 Demo

预计 0.5-1 天。

工作内容：

- mock 数据
- tools schema
- 本地工具函数
- tool-calling 循环
- 系统提示词
- 欢迎语和示例问题
- 手动验证 3-5 条演示路径

### 更强的内部试用原型

预计 2-4 天。

需要增加：

- 更完整订单状态机
- 持久化存储
- 更多异常规则
- 多轮任务状态
- 更细的客户沟通模板

### 生产系统雏形

预计 2-4 周起。

需要增加：

- 真实订单数据接入
- 用户权限
- 消息渠道集成
- 调度和通知机制
- 审计日志
- 失败重试和监控

## 验证标准

实现完成后，至少验证：

1. 启动 Chainlit 应用成功。
2. 查询正常订单能返回摘要和当前状态。
3. 查询风险订单能识别风险并给出建议动作。
4. 查询异常订单能说明异常原因并生成客户话术。
5. 扫描全部订单能列出风险排序。
6. 查询不存在订单不会编造结果。
7. tool calling 不可用时 fallback 路径仍能完成核心演示。

## 推荐实施顺序

1. 增加 mock 数据和工具函数。
2. 增加 tools schema。
3. 实现 tool-calling 循环。
4. 增加系统提示词和欢迎语。
5. 增加兼容性 fallback。
6. 手动验证演示问题。

该顺序可以先让本地业务能力跑通，再接入模型工具调用，降低调试风险。
